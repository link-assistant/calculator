# Case Study: Issue #121 — Spurious `(expression "cos(0)")` alternative for function calls

## Summary

When evaluating expressions with function calls like `cos(0)`, the calculator generates a spurious alternative interpretation `(expression "cos(0)")` alongside the correct LINO form `(cos (0))`. Selecting this alternative causes a parse error:

```
Parse error: Unexpected character '"' at position 12
```

Additionally, the bug report generated by the "Report Issue" button does not include alternative interpretations, making it harder for reporters to communicate the full scope of issues involving ambiguous expressions.

## Environment

- **Version**: 0.11.0
- **URL**: https://link-assistant.github.io/calculator/?q=KGV4cHJlc3Npb24lMjAlMjJjb3MoMCklMjIp
- **Affected expressions**: Any expression containing function calls (`cos(0)`, `sin(pi)`, `sqrt(4)`, `integrate(x^2, x, 0, 3)`, etc.)

## Reproduction Steps

1. Open the calculator and enter `cos(0)`.
2. The result `1` is displayed correctly with LINO `(cos (0))`.
3. An alternative interpretation button `(expression "cos(0)")` is shown.
4. Click the alternative interpretation button.
5. **Result**: Parse error: `Unexpected character '"' at position 12`.

## Timeline / Sequence of Events

1. User enters `cos(0)`.
2. Lexer tokenizes: `cos` -> `Identifier("cos")`, `(` -> `LeftParen`, `0` -> `Number(0)`, `)` -> `RightParen`.
3. Parser recognizes `cos` as a math function followed by `(`, creates `Expression::FunctionCall { name: "cos", args: [Number(0)] }`.
4. `to_lino()` correctly generates `(cos (0))` (space-separated LINO function call syntax).
5. `alternative_lino()` calls `collect_alternatives()` on the expression.
6. **Bug**: The `FunctionCall` branch in `collect_alternatives()` unconditionally generates `(expression "cos(0)")` — a fake "mathematical notation" alternative that wraps the original input in quotes inside an `expression` pseudo-function.
7. This alternative is presented to the user as a clickable button.
8. When clicked, the calculator tries to parse `(expression "cos(0)")` as a LINO expression.
9. The parser encounters the `"` character (which is not valid in LINO syntax) and raises a parse error.

## Root Cause

In `src/types/expression.rs`, the `collect_alternatives()` method (lines 407-416) had a `FunctionCall` match arm that generated a spurious alternative for **every** function call:

```rust
Self::FunctionCall { name, args } if !args.is_empty() => {
    let args_str = args
        .iter()
        .map(|a| a.to_lino_internal(None))
        .collect::<Vec<_>>()
        .join(", ");
    let alt = format!("(expression \"{name}({args_str})\")");
    alternatives.push(alt);
}
```

This code:
1. Creates a string like `(expression "cos(0)")` which is **not valid LINO syntax** — the `"` character is not a valid token in the LINO parser.
2. The `expression` pseudo-function does not exist in the grammar — it was an ad-hoc construct that was never parseable.
3. Unlike other alternatives (precedence regrouping, ambiguous units), this alternative adds **no semantic value** — it just rephrases the same function call in a different notation.

### Secondary Issue: Bug Report Missing Alternatives

In `web/src/utils/reportIssue.ts`, the `generateIssueReport()` function only includes the default `lino_interpretation` in the report body. The `alternative_lino` array (which lists all alternative interpretations) is not included. This means when users report issues involving ambiguous expressions (like `2 + 3 * 4`), the report only shows the default interpretation, not all the alternatives that may be relevant to the bug.

## Solution

### Fix 1: Remove spurious `expression` alternatives (Rust)

Replaced the `FunctionCall` branch in `collect_alternatives()` to only recurse into function arguments (to collect genuine alternatives like ambiguous units), rather than generating the invalid `(expression ...)` wrapper:

```rust
Self::FunctionCall { args, .. } if !args.is_empty() => {
    for arg in args {
        arg.collect_alternatives(alternatives);
    }
}
```

This preserves the ability to detect genuine ambiguities within function arguments (e.g., `cos(19 ton)` where `ton` could be mass or crypto currency) while eliminating the spurious alternative.

### Fix 2: Include alternatives in bug report (TypeScript)

Updated `generateIssueReport()` to include all alternative interpretations when `alternative_lino` has more than one entry:

```typescript
if (state.result.alternative_lino && state.result.alternative_lino.length > 1) {
    sections.push('');
    sections.push(`**Alternative interpretations**:`);
    for (const alt of state.result.alternative_lino) {
        sections.push('```');
        sections.push(alt);
        sections.push('```');
    }
}
```
